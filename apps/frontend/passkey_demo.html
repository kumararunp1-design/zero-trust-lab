Authentication Evolution: Passwords to Passkeys

Traditional (Passwords)
  • Shared secrets (server stores hash)
  • Phishable (user types into fake site)
  • Reuse across sites = credential stuffing
  • MFA helps but adds friction
Modern (Passkeys / FIDO2)
  • Asymmetric keypair (private key never leaves device)
  • Phishing-proof (origin-bound)
  • Unique per site (no credential reuse)
  • Built-in MFA (biometric + possession)
Industry Adoption (2024-2025): Apple, Google, and Microsoft have all implemented passkeys in their platforms. Google reports 50% fewer sign-in issues with passkeys. GitHub and AWS IAM now support passkeys for account login. NIST SP 800-63B-4 (draft) recommends phishing-resistant authenticators.

Interactive WebAuthn Demo

This demo uses the WebAuthn browser API with Chrome's built-in virtual authenticator or your platform authenticator (TouchID, Windows Hello, etc). No physical security key needed.

Step 1: Register a Passkey (navigator.credentials.create)

Browser calls navigator.credentials.create()
Authenticator generates new keypair (ES256)
User verifies with biometric/PIN (user presence)
Public key + attestation returned to server
Server stores public key (never sees private key)

Step 2: Authenticate with Passkey (navigator.credentials.get)

Server sends challenge (random nonce)
Browser calls navigator.credentials.get()
User verifies with biometric/PIN
Authenticator signs challenge with private key
Server verifies signature with stored public key
(Register a passkey first)

Examine what was created during registration and authentication.

Keycloak WebAuthn Integration (Lab Exercise)

Enable WebAuthn in Keycloak Admin Console

Follow these steps to add passkey support to the lab's Keycloak instance:

Step 1: Open Keycloak Admin: http://localhost:8080/admin (admin / admin123)
Step 2: Select realm "zero-trust-lab"
Step 3: Go to Authentication → Flows
Step 4: Select "browser-with-webauthn" flow (pre-configured in realm)
Step 5: Go to Authentication → Bindings, set Browser Flow to "browser-with-webauthn"
Step 6: Login as alice, you'll see option to register a security key
Step 7: In Chrome: DevTools → More Tools → WebAuthn tab → Enable virtual authenticator

Chrome Virtual Authenticator Setup

No physical hardware key needed. Chrome DevTools provides a software authenticator:

1. Open Chrome DevTools (F12)
2. Click "..." menu → More Tools → WebAuthn
3. Check "Enable virtual authenticator environment"
4. Add new authenticator:
   - Protocol: ctap2
   - Transport: internal
   - Supports resident keys: checked
   - Supports user verification: checked
5. Now any WebAuthn API call will use this virtual authenticator

What Happens Under the Hood

// Registration: Server creates challenge, browser creates credential
const credential = await navigator.credentials.create({
    publicKey: {
        challenge: serverChallenge,        // Random bytes from server
        rp: { name: "Zero Trust Lab", id: window.location.hostname },
        user: {
            id: userId,                    // Opaque user handle
            name: "alice@zerotrust.lab",
            displayName: "Alice Admin"
        },
        pubKeyCredParams: [
            { type: "public-key", alg: -7 }  // ES256 (ECDSA w/ SHA-256)
        ],
        authenticatorSelection: {
            authenticatorAttachment: "platform",  // Built-in (TouchID/Hello)
            residentKey: "preferred",             // Discoverable credential
            userVerification: "preferred"         // Biometric if available
        },
        attestation: "none"                // Lab doesn't need attestation
    }
});
// credential.response.getPublicKey() -> sent to server, stored in DB
// Private key NEVER leaves the authenticator

Why Passkeys are Phishing-Proof

Origin Binding: Private key is bound to rp.id (the current domain). A phishing site at "evil.com" cannot trigger the key for this origin.
Challenge-Response: Server sends fresh random challenge each time. Replay attacks impossible since signatures are unique per challenge.
No Shared Secrets: Server only stores public key. Even if the server database is breached, attackers cannot authenticate (no private key to steal).
User Verification: Biometric or PIN required = inherent 2FA (something you HAVE + something you ARE/KNOW) in a single gesture.
Zero Trust Connection: Passkeys enforce "never trust, always verify" at the authentication layer. Combined with short-lived JWTs from Keycloak and per-request OPA policy checks, every access decision is cryptographically verified end-to-end.
